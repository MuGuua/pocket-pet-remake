# 项目规则

## 1. 项目定位

- 本项目是“口袋精灵联机复刻版”的双端仓库，目录固定分为 `backend/` 与 `client/`。
- 当前阶段只围绕 4 个 MVP 核心能力推进：登录与实时连接、世界状态同步、宠物养成与编队、服务端权威战斗结算。
- 未被明确纳入当前阶段的能力，默认不实现，包括但不限于：公会、交易行、邮件、排行榜、跨服、实时 PvP、开放世界实时技能战斗。
- 所有新增实现都必须先判断是否属于当前 MVP 边界；不在边界内的需求，先提示并确认后再动手。
- 实现任何需求前都应该检查原客户端中是否有相关实现，逻辑是否有相似。如果有相似逻辑，建议参考原客户端的实现。
## 2. 总体原则

- 先读文档再修改：开始任务前优先阅读 `backend/docs/architecture.md`、`backend/docs/changelog.md`，必要时补读 `protocol.md`、`message-routing.md`、`task-summary.md`。
- 先复用后新增：优先复用现有模块、场景、协议、控制器和状态结构，避免重复造轮子。
- 最小改动原则：只改与当前需求直接相关的文件、逻辑和资源，不顺手扩散重构，不混入无关优化。
- 保持双端一致：涉及协议、状态字段、消息流转、场景切换、战斗流程的改动，必须同时检查客户端和服务端是否一致。
- 文档与实现一致：如发现文档和代码冲突，以当前运行代码和已有实现链路为准，并在输出中明确说明差异。

## 3. 双端架构约束

- 服务端目录固定在 `backend/`，客户端目录固定在 `client/`，不要混放代码、资源或配置。
- 服务端必须遵循 `transport -> module -> data` 单向依赖，`handler` 只做协议适配和参数转换，不直接承载核心业务逻辑。
- 客户端 UI 不直接发送底层网络包，统一通过 `autoload`、控制器或现有封装入口发起请求。
- 客户端负责表现、输入、UI 和必要的轻量本地反馈；服务端负责最终判定、状态推进、战斗计算、同步广播和防作弊。
- 所有影响结果的动作都按“意图上报”处理，客户端不能伪造最终坐标、伤害、结算结果或战斗状态。

## 4. 服务端规则

- 服务端核心模块以 `auth`、`session`、`player`、`world`、`pet`、`battle`、`bag` 为边界，新增逻辑必须优先归属到已有模块。
- `ws handler` 和 `http handler` 只负责收包、鉴权、调用 service、回包，不写跨模块业务编排。
- 任何世界位置、战斗结果、技能结算、掉落和经验变更，都必须由服务端生成和推送。
- 涉及仓储时优先沿用现有接口与 provider 装配方式，不绕过 `repo` 直接写底层存储。
- 当前仓库同时保留内存版与 PostgreSQL/Redis 骨架，新增数据访问逻辑时要兼容现有抽象，不要把具体存储实现写死到业务层。

## 5. 客户端规则

- 客户端基于 Godot 4 + GDScript，目录与职责按现有结构保持：`autoload/` 管全局能力，`scripts/feature/*` 管业务，`scenes/*` 管场景表现。
- 场景职责要单一，例如登录、主运行态、战斗视图应各自独立，不把多种职责堆进一个场景脚本。
- 小窗口适配是既定约束，UI 设计默认兼容 `320x480`，避免做大面板、重叠遮挡和过度占屏。
- 世界层、战斗层、登录层切换时，优先复用现有状态与网络链路，避免为切场景重建无必要的连接和路由。
- 客户端本地状态只能用于展示和交互组织，不能替代服务端权威状态。

## 6. 协议与同步规则

- 实时通信以 `backend/docs/protocol.md` 和 `backend/server/internal/protocol/` 当前实现为准。
- 当前 WebSocket 消息体是 `JSON`，不是 `protobuf`；若未来切换编码方式，必须先更新协议文档、路由和双端实现，再统一迁移。
- 新增消息号、字段、错误码或推送类型时，必须同步检查：
  - 服务端协议定义与 handler
  - 客户端 `CommandIds`、路由注册与消费逻辑
  - 协议文档说明
- 世界同步和战斗同步是两套独立状态流，新增逻辑时不要把世界状态和战斗状态混在同一套版本推进里。
- 断线重连、重同步、场景切换类需求，优先复用已有 `GameState`、`NetClient`、`world_controller`、`battle_controller` 链路。

## 7. 功能开发规则

- 新功能开发前，先检查当前仓库是否已有相近能力、占位模块或协议编号，优先补全，不另起一套新体系。
- 在已有功能上加能力时，必须保证不破坏现有登录、进入世界、移动、战斗、切场景等主链路。
- 涉及客户端与服务端联动的需求，先明确“谁负责判定、谁负责表现、谁负责同步”，再开始编码。
- 战斗相关改动默认遵循服务端权威，客户端仅提交动作意图和展示推送结果。
- 世界相关改动默认以服务端场景与位置广播为准，客户端只能做位置表现、插值和边界触发。
- 地图、资源、场景类改动应优先接入现有世界流程，不额外引入无关编辑器插件、资源格式或构建步骤。

## 8. 代码风格规则

- 保持现有命名风格、目录风格、缩进和文件组织方式，不引入与现有仓库明显不一致的写法。
- 注释只写必要信息，解释“为什么这样做”或“这里的约束是什么”，不要写低价值注释。
- 单个改动尽量聚焦单一目标，避免把功能开发、重构、格式化、命名清洗混在同一次提交里。
- 默认不新增依赖；确实需要新增依赖时，必须说明必要性、替代方案和影响范围。
- 不为“看起来更优雅”而改动稳定链路，优先保证可读、可维护、可联调。
- 每个变量、常量、方法、函数、公式需要添加详细的注释说明，不需要解释改动的原因。只标注最新代码的说明

## 9. 验证规则

- 完成修改后必须做基础自检，至少检查语法、引用、路由、字段名、场景路径、资源路径是否一致。
- 服务端改动优先运行与改动相关的测试；如果影响面明确，也可补充执行 `go test` 进行验证。
- 客户端改动后要检查 Godot 诊断，确认没有新增脚本错误、场景引用错误或明显运行态问题。
- 涉及双端联动时，要确认请求、响应、推送、状态更新至少在逻辑上闭环，不能只改一端。
- 如无法完成自动验证，需要在结果中明确说明未验证部分、原因和潜在风险。

## 10. 文档维护规则

- 当任务涉及架构边界、协议字段、关键流程或模块职责变化时，应同步更新相应文档。
- 优先维护以下文档的一致性：
  - `backend/docs/architecture.md`
  - `backend/docs/changelog.md`
  - `backend/docs/protocol.md`
  - `backend/docs/message-routing.md`
  - `backend/docs/task-summary.md`
- 若本次任务只是局部实现、未改变项目约束或外部接口，可不额外扩写文档，但不能写出与文档相冲突的实现。

## 11. 禁止事项

- 禁止客户端本地计算最终战斗结果、伪造世界位置或绕过服务端校验。
- 禁止在 `handler`、UI 场景脚本或临时工具脚本里塞入核心业务。
- 禁止在未确认需求的情况下扩展 MVP 边界。
- 禁止为了当前任务顺手改动无关模块、无关资源和无关配置。
- 禁止文档写一套、代码实现另一套，尤其是协议编号、字段结构、消息流转和状态来源。

## 12. 执行口径

- 遇到需求不明确时，先澄清边界、输入输出和归属模块，再修改代码。
- 遇到现有架构与需求冲突时，优先保护现有主链路稳定，并给出最小代价方案。
- 输出方案时，优先说明改动目标、影响文件、验证方式和残余风险。
