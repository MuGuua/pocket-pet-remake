# `kdjl` 原版客户端可参考逻辑梳理

## 1. 目的与边界

- 本文只整理 `/Users/wangzhiwei/study/kdjl` 中**对当前联机复刻版 MVP 有参考价值**的客户端逻辑。
- 当前只关注 4 个 MVP 核心能力：
  - 登录与实时连接
  - 世界状态同步
  - 宠物养成与编队
  - 服务端权威战斗结算
- 以下能力虽然在原版中存在或可见痕迹，但**不纳入当前实现参考**：
  - 公会、交易、邮件、活动跳转、拍照上传、广告页、短信/网页下载
  - 旧时代 WAP/代理联网适配细节
  - 原版服务端驱动 UI 文本协议本身

## 2. 结论

`kdjl` 有值得参考的逻辑，但价值主要在**流程设计**，而不是协议格式或客户端技术实现本身。

最值得保留的有 5 类：

1. 启动到登录成功的前置状态机
2. 登录上下文与本地缓存的最小持久化模型
3. 世界场景、地图/地点面板、战斗场景之间的切换关系
4. 战斗阶段“客户端只选动作与目标，服务端回推状态”的意图模型
5. 统一的消息入口与“当前场景 + 全局 UI”双层分发思路

不建议直接照搬的有 4 类：

1. 文本协议 + 自定义混淆握手
2. 服务端直接下发 `<menu>` / `<input>` 驱动客户端 UI
3. 直连/WAP 代理/网络探测的历史兼容方案
4. 把账号密码等敏感信息长期明文缓存并参与外部跳转

## 3. 逆向入口与可信度

- 主入口：`Heart` -> `O0O0O`
- 登录前场景：`O000O0O`
- 全局 UI/消息解释器：`OO0O`
- 网络引导：`O00O00`
- 主长连接线程：`OOO0O0`
- 世界主场景：`OO00OO`
- 地图/地点列表面板：`OO0000O`
- 战斗场景：`O0OOOOO`

当前可读性判断：

- `O0O0O`、`OO0O`、`O000O0O`、`O00O00`、`OO0000O`、`O0OOOOO` 的核心流程基本可读。
- `OOO0O0` 的协议细节存在一定混淆，但不影响判断“统一发包入口 + 独立网络线程 + 收包后再分发”的总体结构。
- 因此本文结论可作为**流程参考文档**，但不适合作为逐行对抄依据。

## 4. 最值得参考的逻辑

### 4.1 启动与登录前状态机

原版不是“进游戏即登录”，而是一个清晰的启动前状态机：

1. `Heart.startApp()` 创建主画布 `O0O0O`
2. `O0O0O.run()` 完成首屏、资源加载、本地配置读取、UI 管理器初始化
3. 当前主场景默认切到 `O000O0O`
4. `O000O0O` 负责欢迎页、账号输入、网络设置、角色创建、角色选择
5. 登录成功后，再由全局解释器 `OO0O` 收到 `<log_suc>` 并切到正式世界场景 `OO00OO`

对当前项目的参考价值：

- 当前项目已经拆出了 `login_scene` 与 `main/world`，方向是对的。
- 后续可以继续把“登录前状态”显式化，而不是把所有前置流程塞进单个按钮回调里。
- 建议保留以下独立阶段：
  - 账号登录
  - WebSocket 鉴权
  - 进入世界
  - 角色/编队初始化校验
  - 失败后的返回登录页或重试

这部分更像“场景状态机设计参考”，非常值得保留。

### 4.2 登录上下文的最小本地持久化

原版会保存一组最小但高价值的登录上下文：

- `usr` / `pwd`
- `line` / `server`
- `lastID` / `lastUsr`
- `cityID`
- `#maps`
- `setting`

其中真正值得今天参考的不是“保存密码”，而是下面这些信息组织方式：

- 最近一次登录使用的线路/服务器
- 最近一次角色或玩家标识
- 地图缓存版本或地图资源持有情况
- 客户端配置项与展示开关

对当前项目的参考价值：

- `GameState` 之外，可以考虑补一个明确的“本地会话上下文”概念。
- 建议只持久化安全且必要的数据，例如：
  - 最近账号
  - 最近服务器/环境
  - 最近玩家 ID
  - 已下载地图版本或资源版本
  - 显示设置、战斗表现开关
- 不建议沿用原版的明文密码缓存策略。

### 4.3 统一命令入口与双层分发

原版的 `O0O0O.O00O000(String)` 虽然底层很旧，但设计点很清楚：

- 所有命令统一从一个入口进入
- 再根据前缀决定：
  - 本地立即执行
  - 读缓存
  - 发给网络线程
- 网络收包后并不是直接操作场景，而是先进入统一入站队列
- 每帧由“当前场景 + 全局 UI 管理器”共同消费

这套思路和当前项目的方向高度一致：

- 当前项目已有 `App.gd`
- 网络层已有 `NetClient.gd`
- 分发层已有 `MessageRouter.gd`
- 表现层已有 `world_controller.gd` / `battle_controller.gd` / `pet_controller.gd`

对当前项目的明确启发是：

- 继续坚持“UI 不直接发底层包”
- 继续坚持“控制器只处理自己关心的消息”
- 后续新增宠物/编队/战斗消息时，统一从控制器入口发送，不要从场景脚本四处散发

### 4.4 世界场景、地图面板与战斗场景的切换关系

原版世界层的结构不是把所有玩法揉在同一界面，而是：

- `OO00OO` 作为正式世界主场景
- 收到地图类数据时，交给 `OO0000O` 这样的地点/地图列表面板处理
- 收到战斗切入指令时，再切到 `O0OOOOO` 战斗场景
- 战斗结束收到 `<r>walk` / `<r>follow` 等指令后，再返回世界场景

从 `OO00OO` 可见几个清晰的状态变化：

- `<r>walk`：恢复普通行走
- `<r>follow`：进入跟随某目标的移动状态
- `<r>combat`：开始战斗切入表现
- `<r>rombat`：真正实例化战斗场景并切换

对当前项目的参考价值：

- 世界层和战斗层分离是正确方向，不应混成一个状态树。
- 战斗切入应分成两步：
  - 服务端确认开始战斗
  - 客户端切换显示层并锁定世界输入
- 战斗结束后应恢复世界层而不是重建整条网络链路。

这与当前 `main.gd` 中“挂载 battle_scene 而不是整树切场景”的做法一致，可以继续沿用。

### 4.5 地图/地点选择面板的“图结构驱动”思路

`OO0000O` 不是简单的文本列表，它会解析一段结构化数据，用于表达：

- 地点集合
- 当前地点
- 目标地点
- 节点坐标
- 节点说明文本
- 某些地点的可达/未探索状态

用户在地图面板里做的事情，本质上不是“自己改坐标”，而是：

1. 选择一个目标地点
2. 客户端把地点标识上报给服务端
3. 服务端决定能否移动、切图或传送

对当前项目的参考价值：

- 当前 MVP 的世界移动已经是服务端权威，这一点要继续保持。
- 后续如果加地图切换、传送点、城镇入口，不建议让客户端直接算最终位置。
- 更合适的方式是：
  - 客户端只上报“想去哪个点/哪个门/哪个场景入口”
  - 服务端校验可达性、前置条件、资源版本
  - 服务端回世界重同步或场景切换结果

### 4.6 战斗动作的“意图上报”模型

`O0OOOOO` 最值得参考的不是旧 UI，而是动作建模方式。客户端在战斗里主要做 4 件事：

- 选择普通攻击
- 选择技能
- 选择物品
- 选择逃跑或自动战斗

它发送的是动作意图，例如：

- `attack `
- `peta `
- `escape`
- `f9`

同时，客户端还会等待服务端下发战斗状态相关消息，例如：

- `<r>skill`
- `<r>item`
- `<r>ownpet`
- `<r>pets`
- `<r>auto`
- `<r>endfight`

从职责划分上看，原版已经体现出下面这条重要原则：

- 客户端负责“选择什么动作、选谁作为目标、怎么展示”
- 服务端负责“动作是否合法、谁先出手、结果是什么、战斗何时结束”

这与当前项目的服务端权威战斗约束完全一致，是最值得吸收的核心逻辑之一。

### 4.7 战斗内目标选择与自动战斗的边界

从 `O0OOOOO` 可以看出，原版战斗客户端会维护：

- 当前可选目标集合
- 当前己方出战宠/单位
- 技能列表
- 物品列表
- 自动战斗开关

但这些状态的作用主要是**组织输入和展示**，而不是直接结算结果。

对当前项目的参考价值：

- `battle_controller.gd` 后续可以继续只保存：
  - 当前战斗快照
  - 可用技能列表
  - 当前行动方
  - 当前可选目标
  - 自动战斗开关
- 不应在客户端本地预演伤害、命中、死亡或经验结算。

### 4.8 角色、宠物、技能三个层次的区分

虽然原版命令很混淆，但战斗代码已经表现出三层对象边界：

- 角色/单位
- 宠物
- 技能/物品动作

特别是这些字段很有参考意义：

- 当前己方出战宠 `ownpet`
- 技能列表 `skill`
- 宠物技能列表 `pets`
- 物品列表 `item`

这说明原版并不是把“玩家”和“宠物战斗单位”混成一个对象。

对当前项目的参考价值：

- 当前服务端 `player`、`pet`、`battle` 拆模块是正确的。
- 后续补宠物编队时，建议明确区分：
  - 玩家拥有的宠物实例
  - 当前编队
  - 当前出战宠
  - 战斗中的临时单位快照

## 5. 不建议直接参考的逻辑

### 5.1 协议格式本身

原版协议具备这些特征：

- 文本命令为主
- 自定义握手
- 长连接线程自行混淆/轻加密
- 文本消息和二进制资源块混发

这些内容只适合用来理解“当年客户端如何组织消息”，不适合迁移到当前项目。

当前项目应继续以现有实现为准：

- HTTP 登录
- WebSocket 实时通信
- 当前消息体按项目现状使用 `JSON`
- 路由、字段与命令号以现仓库实现和文档为准

### 5.2 服务端驱动 UI 文本协议

原版大量使用：

- `<menu>`
- `<input>`
- `<pmg>`
- `<qmsg>`

由服务端直接控制客户端界面内容和行为。

这种方式在今天的问题很明显：

- UI 可维护性差
- 客户端类型安全弱
- 很难做现代化状态管理
- 很难和 Godot 组件化界面对齐

因此只能参考“流程拆分”，不应参考“让服务端拼 UI 文本”这一做法。

### 5.3 WAP/代理联网适配

`O00O00` 和 `O00OOOO` 中的网络探测、代理头、线路切换，是 J2ME 年代的环境兼容方案。

对当前项目而言：

- 没有直接实现价值
- 只能抽象保留“失败重试、入口切换、环境配置持久化”的思想

### 5.4 敏感信息处理方式

原版会长期缓存账号密码，甚至在某些跳转逻辑中带上账号信息。

这一部分应明确视为反面案例，不可复用。

## 6. 对当前项目的落地建议

### 6.1 登录与连接

可参考点：

- 登录前状态机拆分
- 最近登录上下文持久化
- 网络失败后的明确重试阶段

不参考点：

- 原版探网/WAP 代理
- 原版文本登录协议

建议后续落点：

- `client/scripts/auth/login_scene.gd`
- `client/autoload/app.gd`
- `client/autoload/game_state.gd`
- 服务端 `auth` / `session` 模块

### 6.2 世界同步

可参考点：

- 世界场景和地图/地点面板分层
- 场景切换与战斗切换分开
- 客户端只提交移动/交互意图

不参考点：

- 原版地图资源下载协议细节
- 原版世界层混合二进制块与文本协议的做法

建议后续落点：

- `client/scripts/feature/world/world_controller.gd`
- `backend/server/internal/module/world`

### 6.3 宠物与编队

可参考点：

- 玩家宠物实例与当前出战宠分离
- 技能列表、物品列表、当前出战状态分离

不参考点：

- 原版面向 J2ME 菜单的 UI 组织方式

建议后续落点：

- `client/scripts/feature/pet/pet_controller.gd`
- `backend/server/internal/module/pet`
- `backend/server/internal/module/player`

### 6.4 战斗

可参考点：

- 战斗动作只提交意图
- 客户端维护目标选择与展示状态
- 自动战斗只是模式开关，不是本地结算
- 战斗结束后恢复世界显示层

不参考点：

- 原版特效下载、旧战斗资源格式
- 原版文本战斗命令名

建议后续落点：

- `client/scripts/feature/battle/battle_controller.gd`
- `client/scenes/battle/`
- `backend/server/internal/module/battle`

## 7. 建议优先吸收的实现顺序

如果后续继续参考原版实现，优先级建议如下：

1. 登录前状态机与最近登录上下文
2. 世界层与战斗层切换约束
3. 宠物实例 / 编队 / 出战宠三层模型
4. 战斗中的可选目标、技能、物品、自动战斗状态
5. 地图入口/传送点的“意图上报 + 服务端校验”链路

## 8. 一句话结论

`kdjl` 最值得参考的是**流程边界和状态组织**，不是旧协议、旧 UI 技术或旧联网手段。  
对于当前项目，最有价值的吸收方向是：

- 登录前状态机
- 世界与战斗场景分层
- 宠物实例、编队、出战宠分离
- 客户端只提交战斗意图
- 服务端负责最终状态推进与结算
