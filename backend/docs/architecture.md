# 口袋精灵联机复刻版架构草案

## 目标边界

第一版只解决 4 个核心问题：
- 登录与实时连接
- 世界状态同步
- 宠物养成与编队
- 服务端权威战斗与结算

第一版明确不做：
- 公会、交易行、邮件、排行榜
- 跨服与多进程拆分
- 实时 PvP
- 开放世界实时技能战斗

## 总体架构

- 客户端：Godot 4 + GDScript
- 服务端：Go 单体服 + WebSocket + HTTP
- 通信：`HTTP(S)` 登录，`WSS` 实时消息，消息体使用 `Protobuf`
- 持久化：PostgreSQL
- 短时状态：Redis

## 设计原则

- 客户端只负责表现、输入、UI 和本地轻预测
- 服务端负责最终判定、同步广播、战斗结算、防作弊
- 所有影响结果的动作都以“意图”上报，而不是客户端直接提交结果
- 世界同步和战斗同步分离，分别维护自己的状态版本号

## 服务端模块

### auth
- HTTP 登录
- JWT 签发
- `ws_token` 签发与消费

### session
- WebSocket 会话绑定
- 心跳管理
- 顶号与断线重连

### player
- 玩家角色主数据
- 当前场景、位置、等级、货币
- 编队引用与基础属性汇总

### world
- 场景管理
- AOI 广播
- 玩家移动合法性校验
- 野怪遭遇

### pet
- 宠物模板读取
- 玩家宠物实例
- 宠物成长与编队

### battle
- 战斗房间
- 回合状态机
- 行为合法性校验
- 伤害、抓宠、掉落、经验结算

### bag
- 背包与道具
- 道具增减
- 道具使用校验

### protocol
- 包头编解码
- 消息号映射
- protobuf 消息编解码
- checksum 校验

## 客户端模块

### autoload
- `App.gd`：启动初始化
- `HttpClient.gd`：HTTP 登录与刷新
- `NetClient.gd`：WSS 建连、收发、心跳、重连
- `MessageRouter.gd`：按 `cmd` 路由到业务控制器
- `GameState.gd`：全局状态快照

### feature/world
- 世界场景控制
- 输入转移动意图
- 实体表现与插值

### feature/battle
- 战斗 UI
- 技能、目标、切宠输入
- 战斗事件表现

### feature/pet
- 宠物列表
- 宠物详情
- 编队管理

### feature/bag
- 背包列表
- 道具使用

## 强约束

- `transport -> module -> data` 单向依赖
- `ws handler` 不直接写业务逻辑
- UI 不直接发送网络包，统一通过控制器或网络层
- 战斗结果只看服务端推送
- 世界位置最终以服务端广播为准

## MVP 实施顺序

1. 登录、`ws_token`、WebSocket 会话
2. 进入世界、场景快照、AOI、移动同步
3. 宠物实例、宠物列表、编队
4. PvE 战斗状态机
5. 背包、抓宠球、药品
6. 断线重连、限流、持久化优化
