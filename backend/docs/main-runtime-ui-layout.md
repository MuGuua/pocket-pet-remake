# 主运行态 UI 结构设计

## 1. 文档目的

- 本文用于把登录后的客户端主运行态 UI 结构固定下来，作为后续世界地图、战斗场景和常驻 HUD 的统一实现口径。
- 本文只覆盖当前 MVP 范围内真正会用到的能力：
  - 登录与实时连接
  - 世界状态同步
  - 宠物养成与编队
  - 服务端权威战斗结算
- 本文不扩展原版里出现但当前未纳入 MVP 的功能，包括但不限于：
  - 商城
  - 邮件
  - 排行榜
  - 任务系统完整实现
  - 聊天频道完整实现
  - 活动跳转页

## 3. 核心结论

- 登录成功后，客户端应进入一个常驻主运行态，而不是在世界、战斗、UI 面板之间频繁整树切场景。
- 主运行态拆成两层：
  - 上部游戏显示区
  - 下部常驻 HUD 区
- 世界地图、地图切换、战斗场景、过场动画都只发生在上部游戏显示区。
- 下部常驻 HUD 区在登录后持续存在，不跟随地图镜头移动，也不因进入战斗而销毁。
- 下部 HUD 先承接当前 MVP 内真正需要的状态显示和操作入口，超范围入口只保留设计占位，不进入本轮实现。

## 4. 设计目标

本结构需要同时满足下面 5 个目标：

1. 保持 `main.tscn` 为登录后的唯一常驻根场景
2. 保持世界层与战斗层分离，但共用同一个主运行态根容器
3. 让地图切换、战斗切换不影响底部常驻 UI 的生命周期
4. 兼容 `320x480` 小窗口约束
5. 为后续补操作面板、宠物入口、背包入口保留明确挂点

## 5. 结构总览

推荐保持如下结构：

```text
Main
  ├─ GameplayArea
  │   ├─ WorldMount
  │   └─ BattleMount
  ├─ PetController
  ├─ BattleController
  ├─ BagController
  └─ UiLayer
      ├─ HudRoot
      │   ├─ HudBackground
      │   ├─ HudShade
      │   ├─ TopPanel
      │   ├─ BottomPanel
      │   └─ FutureActionPanel
      └─ TransitionOverlay
```

对应当前已实现部分：

- `GameplayArea`：上部游戏显示区
- `WorldMount`：世界层挂载点
- `BattleMount`：战斗层挂载点
- `HudRoot`：下部常驻 HUD 根容器
- `HudBackground`：底部固定背景图
- `TopPanel`：当前状态信息区
- `BottomPanel`：当前日志区
- `TransitionOverlay`：统一转场遮罩

`FutureActionPanel` 目前只是概念挂点，后续如果要把底部做成正式操作区，应优先接在这里，而不是再发明第二套根容器。

## 6. 分区职责

### 6.1 上部游戏显示区

上部区域只负责游戏主内容表现：

- 世界地图显示
- 本地角色与附近实体
- 地图门区切换表现
- 战斗场景显示
- 战斗结果表现
- 切图和切战斗时的必要动画

上部区域不负责：

- 长期状态信息展示
- 常驻日志输出
- 底部菜单背景
- 功能区按钮布局

一句话约束：

- 上部区域负责“游戏在发生什么”
- 不负责“玩家长期操作面板长什么样”

### 6.2 下部常驻 HUD 区

下部区域负责登录后持续存在的非镜头层 UI：

- 当前连接状态
- 当前场景摘要
- 玩家摘要信息
- 当前可用快捷操作入口
- 常驻日志或提示输出
- 底图与装饰性框体

下部区域不负责：

- 地图镜头跟随
- 世界实体绘制
- 战斗技能动画播放
- 权威位置和战斗结果计算

一句话约束：

- 下部区域负责“玩家怎么读状态、怎么发操作”
- 不直接承载游戏世界内容本身

## 7. 生命周期口径

### 7.1 登录成功后

- 进入 `main.tscn`
- 挂载世界层
- 初始化消息路由
- 初始化 HUD
- 请求进入世界

### 7.2 地图切换时

- 保持 `main.tscn` 常驻
- 保持 `HudRoot` 常驻
- 只在 `WorldMount` 内更换当前地图资源
- 如需加载遮罩，优先复用统一 `TransitionOverlay` 或世界层自己的轻量遮罩

### 7.3 进入战斗时

- 保持 `main.tscn` 常驻
- 保持 `HudRoot` 常驻
- 隐藏 `WorldMount`
- 显示 `BattleMount`
- 按战斗状态刷新下部 HUD 的文本或按钮状态

### 7.4 战斗结束时

- 卸载 `BattleMount` 当前战斗场景
- 恢复 `WorldMount`
- 继续沿用同一套底部 HUD
- 继续沿用现有网络链路和控制器

## 8. 当前 MVP 下底部 HUD 应包含什么

当前阶段只建议保留 4 类真实内容：

### 8.1 运行状态区

对应当前已存在：

- 连接状态
- 当前场景
- 玩家信息

这块是必须项，因为它直接服务于当前 MVP 联调和世界同步验证。

### 8.2 世界交互区

对应当前已存在或可继续扩展：

- 挑战附近 NPC
- 进入/退出地图提示
- 切图失败提示

这块是世界同步与战斗入口的最小交互区。

### 8.3 战斗操作区

当前建议口径：

- 战斗主要操作仍在上部战斗视图内呈现
- 下部 HUD 可只做状态摘要或模式切换提示
- 若后续把技能按钮下沉到底部，也应复用同一个 `HudRoot`，而不是额外做一个独立战斗根 UI

### 8.4 宠物与背包入口区

当前建议先保留挂点，不急于扩实现：

- 宠物列表入口
- 编队入口
- 背包入口

这些入口属于当前 MVP 边界内，但是否立即做成固定按钮，需要以后按真实需求逐步接入。

## 9. 当前 MVP 下不应进入实现的部分

原版底部区域里可以看到很多入口分组，但当前项目不应直接照搬为真实功能：

- 商城
- 频道
- 任务完整系统
- 辅助完整系统
- 聊天完整系统

可以保留两种处理方式：

1. 完全不显示
2. 只在设计文档中预留占位，不在当前代码中接按钮逻辑

推荐当前阶段使用第 1 种，避免误导功能边界。

## 10. 状态驱动规则

下部 HUD 必须按当前运行态切换自身显示内容，但它的节点生命周期不变。

推荐分 4 种状态：

### 10.1 未进入世界

- 显示连接中、初始化中、进入世界中等提示
- 不显示地图相关操作
- 不显示战斗相关操作

### 10.2 世界态

- 显示当前场景、当前位置、附近实体摘要
- 显示世界交互按钮
- 显示宠物/编队/背包等后续可接入入口

### 10.3 战斗态

- 保持 HUD 根容器存在
- 隐藏不适用的世界交互按钮
- 显示战斗摘要、回合、当前出战宠等信息
- 如有必要显示战斗专属按钮，也应在此态切换显示

### 10.4 断线或返回登录

- 允许显示错误信息
- 播放统一淡出
- 回到登录场景后销毁整个主运行态

## 12. 推荐演进顺序

后续建议按下面顺序落地：

1. 先固定主运行态分层和节点职责
2. 再把当前调试信息区整理成正式 HUD 信息区
3. 再补宠物、编队、背包入口
4. 最后再做战斗态和世界态的按钮组切换

不要反过来先做大量按钮和样式，再回头改根结构。

## 13. 一句话实施口径

登录后客户端只进入一个主运行态：

- 上部跑世界、战斗、动画
- 下部常驻 HUD 和操作区
- 切图与切战斗只换上部显示层
- 下部生命周期保持稳定
